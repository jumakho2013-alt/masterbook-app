# Аудит MasterBook — 16 апреля 2026

## TL;DR

Проект на твёрдом MVP-уровне: чистая архитектура (Zustand + AsyncStorage + Supabase auth), полный feature-set для релиза (Liquid Glass, биометрия, удаление аккаунта, экспорт, rate-limit, Zod-валидация), 44 зелёных unit-теста и корректные EAS-профили.

Три группы реальных рисков:
1. **Source-of-truth раздвоен** — AsyncStorage и Supabase таблицы готовы, но не синхронизируются. Масштабирование к >200 записям / второму устройству сломается.
2. **Store-compliance пробелы** — `eas.json` содержит placeholder `ascAppId`/`appleTeamId`; `.env.local` с `EXPO_PUBLIC_DEV_PREVIEW=1` обходит auth — нужно гарантировать что не попадёт в production builds; нет `LICENSE` файла (README говорит "Proprietary").
3. **Accessibility на троечку** — touch-targets 32pt в chip-ах, `textTertiary` не проходит WCAG AA (3.8:1 light, 2.8:1 dark), нет `accessibilityLabel` на многих иконках, нет `reduce motion` guard.

В первую очередь закрыть: (a) hardcoded dev bypass-контроль, (b) контраст цветов, (c) чистку ScrollView→FlatList для Финансов, (d) вынести дублирующиеся time-утилиты.

## Критические (must fix)

### C1. Dev bypass может попасть в production
**Файлы:**
- [app/index.tsx:45–50](app/index.tsx) — `if (__DEV__ && process.env.EXPO_PUBLIC_DEV_PREVIEW === '1') return <Redirect href="/(tabs)" />`
- [.env.local:9](.env.local) — `EXPO_PUBLIC_DEV_PREVIEW=1`

**Риск:** `__DEV__=false` в release-билдах отсечёт bypass, но если кто-то сделает `eas build --profile development` в CI и случайно загрузит в TestFlight — auth пропадёт.

**Корректный фикс:** в `eas.json` production-профиле явно поставить `env.EXPO_PUBLIC_DEV_PREVIEW: "0"` (EAS env переопределяет `.env.local`). Плюс CI-проверка: падать на `grep "EXPO_PUBLIC_DEV_PREVIEW=1" .env.local.example` после release-build.

---

### C2. `eas.json` — placeholder учёток
**Файл:** [eas.json:60–61](eas.json)
```json
"ascAppId": "REPLACE_WITH_APP_STORE_CONNECT_APP_ID",
"appleTeamId": "REPLACE_WITH_APPLE_TEAM_ID"
```

**Риск:** `eas submit` упадёт на первом же вызове. Это не блокер билда, но блокер submission. Документировано в `APP_STORE_METADATA.md`.

**Корректный фикс:** после получения Apple Developer — заменить значения. Альтернатива: через `eas secret` + `$EAS_SUBMIT_APPLE_TEAM_ID` в eas.json.

---

### C3. Контраст `textTertiary` не WCAG AA
**Файл:** [src/theme/colors.ts](src/theme/colors.ts)
- Light: `textTertiary: '#A0A0B0'` на `background: '#FAF9F7'` ≈ **3.8:1** (требуется 4.5:1 для normal, 3:1 для large).
- Dark: `textTertiary: '#5A5A70'` на `background: '#13131A'` ≈ **2.8:1** — fail даже для large.

**Где используется:** subtitle-ы menu-items, placeholder-ы Input-ов, empty-state sub-тексты, «и ещё 3...» в sleeping clients.

**Риск:** люди с low vision не прочитают. Apple / Google не блокируют submission, но отзывы в рейтинге будут.

**Корректный фикс:** Light `textTertiary` → `#7A7A8A` (4.6:1), Dark `textTertiary` → `#8A8A9E` (4.5:1). Проверить оба режима после.

### C4. Дублирование time-утилит (корень багов в будущем)
**Файлы:**
- [app/appointment/new.tsx:31–61](app/appointment/new.tsx) — `generateTimeSlots`, `timeToMinutes`, `addMinutes`
- [app/appointment/[id].tsx:38–57](app/appointment/[id].tsx) — та же тройка, `addMinutes` переименована в `addMinutesToTime`
- [app/(tabs)/index.tsx:24–28](app/(tabs)/index.tsx) — ещё `timeToMinutes`

**Риск:** когда начнём исправлять логику слотов (например, поддержка часовых поясов, 15-мин шагов вместо 30), баг в одном месте не проявится в другом. Классический источник production-багов.

**Корректный фикс:** создать `src/utils/time.ts` с полным набором (`generateTimeSlots`, `timeToMinutes`, `addMinutes`, `nowHourMinutes`) и импортировать из всех трёх файлов. Добавить unit-тесты.

## Важные (should fix)

### V1. ScrollView вместо FlatList на странице Финансов
**Файл:** [app/(tabs)/finances.tsx:143](app/(tabs)/finances.tsx)
- Основной экран с потенциально растущим набором транзакций рендерится в `ScrollView`. При 50+ записях — janky scrolling.

### V2. `expo-image` для фото работ
**Файл:** [app/appointment/[id].tsx:317–323](app/appointment/[id].tsx)
- Используется обычный `<Image>` без FastImage / `expo-image` → при 5+ фото при скролле декомпрессия каждый раз.

### V3. Zustand селекторы без `shallow`
**Пример:** [app/(tabs)/index.tsx:41–43](app/(tabs)/index.tsx)
```tsx
const appointments = useAppointmentStore((s) => s.appointments);
const clients = useClientStore((s) => s.clients);
const services = useServiceStore((s) => s.services);
```
- Любое изменение стора триггерит rerender главного экрана, даже если затронутый массив не изменился. Архитектурно хрупко.

### V4. Accessibility-лейблы на иконках в списках
- [src/components/AppointmentCard.tsx:89](src/components/AppointmentCard.tsx) — ChevronRight без label
- [src/components/ClientRow.tsx:56](src/components/ClientRow.tsx) — ChevronRight без label
- [app/(tabs)/index.tsx:155–156](app/(tabs)/index.tsx) — pulse dot как содержательный элемент без описания

**Фикс:** заменить иконки на `accessibilityElementsHidden` или вся карточка → `accessibilityRole="button"` + `accessibilityLabel="{client.name}, {service.name}, открыть детали"`.

### V5. Reduce motion не респектится
**Файлы:** все `entering={FadeInDown}`, все `withSpring`, `Animated.View` в табах, `AppointmentCard`, `ClientRow`, `IconButton`.

**Фикс:** обёртка `useReduceMotion()` hook через `AccessibilityInfo.isReduceMotionEnabled()`, если `true` → `entering={undefined}` или мгновенное spring.

### V6. Touch-targets < 44pt
- [app/(tabs)/clients.tsx:246](app/(tabs)/clients.tsx) — filter/sort chips `height: 32`
- [app/(tabs)/index.tsx:269](app/(tabs)/index.tsx) — filter chip (не фиксированная высота, но padding 8 × text ≈ 30pt)

**Фикс:** `minHeight: 44` + `hitSlop={{ top: 6, bottom: 6 }}` чтобы визуально остались тонкими.

### V7. Нет тестов на критичные flows
- `src/lib/deleteAccount.ts` — сложный ordering (notifications → RPC → signOut → AsyncStorage.clear → reset). Если один из шагов упадёт и не вернёт ошибку — данные не сотрутся, аккаунт останется. **Нужны mock-тесты.**
- `src/lib/notifications.ts:scheduleAppointmentReminder` — парсит даты вручную. Граничные случаи (полночь, високосный год, DST) — не покрыты.
- `src/stores/useAppointmentStore.ts:setStatus` — имеет side-effect отмены нотификации. Никаких тестов.

### V8. Session tokens в AsyncStorage на Android
**Файл:** [src/lib/supabase.ts:9](src/lib/supabase.ts)
```ts
storage: AsyncStorage,
```
- iOS: AsyncStorage оборачивается в Keychain (auto-encrypted). OK.
- Android: AsyncStorage = SharedPreferences → XML в открытом виде. Root-пользователь прочитает.

**Фикс:** `react-native-encrypted-storage` или `expo-secure-store` для Android (с iOS fallback на стандартный AsyncStorage).

## Рекомендации на будущее

1. **Supabase sync** — приоритет #1 для v1.1. Сейчас данные теряются при переустановке. Решение: после auth — `supabase.from(table).select()` → merge в Zustand stores. Offline-queue на AsyncStorage (уже спланировано в README).
2. **Автогенерация типов Supabase** — `supabase gen types typescript` → `src/types/supabase.ts`, импортировать в stores. Избавит от drift между `supabase-schema.sql` и `src/types/client.ts`.
3. **E2E через Maestro** — дешевле чем Detox, декларативный YAML. Пяток критичных flows: регистрация → клиент → запись → экспорт → удаление.
4. **Feature flags** — сейчас `EXPO_PUBLIC_DEV_PREVIEW` — единственный флаг. Когда появятся A/B тесты (онлайн-запись, PRO-подписка) — нужна система (ConfigCat, LaunchDarkly, или свой JSON в Supabase).
5. **Sentry / crash reporter** — сейчас ошибки видны только в `ErrorScreen` (пользователь может скопировать). Production нужна автоматическая отправка.
6. **Analytics** — 0 событий трекается. Не плохо для privacy, но невозможно понять что люди делают.
7. **PRO-подписка через IAP** — декларирована в `profile.tsx` как баннер, но реально не реализована. Требует `react-native-iap` или `expo-in-app-purchases` + StoreKit config.
